# SOLID 원칙 핵심 요약 및 예제

### 1. SRP (단일 책임 원칙)
* **개념:** 클래스는 단 하나의 책임(변경될 이유)만 가져야 함.
* **실생활 예제:** 빵집에서 제빵사(BreadBaker)는 빵만 구워야 함. 재고 관리, 청소, 고객 응대 등을 혼자 다 하면 안 됨. 각각의 역할을 분리해야 함.

### 2. OCP (개방/폐쇄 원칙)
* **개념:** 확장에는 열려(Open) 있고, 수정에는 닫혀(Closed) 있어야 함.
* **실생활 예제:** 결제 시스템에 'PayPal 결제'를 추가할 때, 기존의 '신용카드 결제' 코드를 뜯어고치는 것이 아니라 새로운 PayPal 클래스를 추가해서 확장함.

### 3. LSP (리스코프 치환 원칙)
* **개념:** 자식 클래스는 부모 클래스의 역할을 완벽히 대체할 수 있어야 함. (부모의 규칙을 깨면 안 됨)
* **실생활 예제:** 직사각형(부모)을 상속받아 정사각형(자식)을 만들면 안 됨. 직사각형은 가로/세로를 따로 변경할 수 있지만, 정사각형은 같이 변경되어야 하므로 부모의 동작 방식을 위반하게 됨.

### 4. ISP (인터페이스 분리 원칙)
* **개념:** 사용하지 않는 기능(메서드)을 강제로 구현하게 만들면 안 됨. 인터페이스는 작고 구체적으로 쪼개야 함.
* **실생활 예제:** 단순 프린터 기기에게 '스캔', '팩스' 기능까지 포함된 거대한 복합기 인터페이스를 상속받게 하면 안 됨.

### 5. DIP (의존성 역전 원칙)
* **개념:** 고수준 모듈(메인 로직)이 저수준 모듈(세부 구현)에 직접 의존해선 안 되며, 둘 다 '추상화(인터페이스)'에 의존해야 함.
* **실생활 예제:** 전등 스위치가 특정 전구 브랜드를 직접 알고(결합) 있으면 안 됨. 어떤 전구가 오든 켜고 끌 수 있는 '스위치 규격'을 만들어야 함.


## 예제 코드

```JAVA
/**
 * [Bad Case] SRP 위반
 * 제빵사가 빵 굽기, 청소, 결제를 모두 관리함.
 * 청소 로직이 바뀌거나 결제 수단이 추가될 때 '제빵사' 클래스를 수정해야 함.
 */
class BreadBaker {
    void bakeBread() {
        System.out.println("빵을 굽습니다.");
    }

    void cleanShop() {
        System.out.println("매장을 청소합니다.");
    }

    void processPayment() {
        System.out.println("결제를 처리합니다.");
    }
}

/**
 * [Good Case] SRP 준수
 * 각 역할에 맞게 클래스를 분리하여, 
 * 변경이 필요할 때 해당 클래스만 수정하면 됨.
 */
class Baker {
    void bake() {
        System.out.println("빵을 굽습니다.");
    }
}

class Cleaner {
    void clean() {
        System.out.println("매장을 청소합니다.");
    }
}

class Cashier {
    void process() {
        System.out.println("결제를 처리합니다.");
    }
}
```


```JAVA
/**
 * [Bad Case] OCP 위반
 * 새로운 결제 방식(NaverPay)이 추가될 때마다 
 * processPayment 내부의 if-else 문을 계속 수정해야 함. (수정에 열려 있음)
 */
class PaymentService {
    void processPayment(String type) {
        if (type.equals("CreditCard")) {
            System.out.println("신용카드 결제 처리");
        } else if (type.equals("PayPal")) {
            System.out.println("페이팔 결제 처리");
        }
        // 네이버페이가 추가된다면? 또 이 코드를 고쳐야 함.
    }
}

/**
 * [Good Case] OCP 준수
 * 인터페이스를 활용하여 확장은 열려 있고, 
 * 기존 PaymentService의 코드는 수정할 필요가 없음. (수정에 닫혀 있음)
 */
interface Payment {
    void process();
}

class CreditCardPayment implements Payment {
    public void process() { System.out.println("신용카드 결제 처리"); }
}

class PayPalPayment implements Payment {
    public void process() { System.out.println("페이팔 결제 처리"); }
}

// 새로운 결제 수단 추가 시 기존 코드 수정 없이 클래스만 생성하면 됨
class NaverPayPayment implements Payment {
    public void process() { System.out.println("네이버페이 결제 처리"); }
}

class PaymentService {
    void processPayment(Payment payment) {
        payment.process(); // 어떤 결제가 들어오든 이 로직은 변하지 않음
    }
}
```

```JAVA
/**
 * [Bad Case] LSP 위반
 * 정사각형은 직사각형의 일종이라 생각해서 상속받았으나,
 * 부모의 메서드(가로/세로 독립 설정)를 마음대로 바꿔버려 부모의 기대를 저버림.
 */
class Rectangle {
    protected int width;
    protected int height;

    public void setWidth(int width) { this.width = width; }
    public void setHeight(int height) { this.height = height; }
    public int getArea() { return width * height; }
}

class Square extends Rectangle {
    @Override
    public void setWidth(int width) {
        this.width = width;
        this.height = width; // 정사각형이라서 세로까지 강제로 바꿈
    }

    @Override
    public void setHeight(int height) {
        this.width = height; // 정사각형이라서 가로까지 강제로 바꿈
        this.height = height;
    }
}
// 결과적으로 Rectangle 타입으로 Square를 쓸 때 가로만 바꿨는데 
// 세로까지 바뀌어버리면 Rectangle을 사용하는 사용자 입장에서 당황스러운 상황이 됨.


/**
 * [Good Case] LSP 준수
 * 두 도형의 공통점(넓이 구하기)만 추출하여 인터페이스로 정의함.
 * 이제 부모(Shape)의 역할을 자식이 수행할 때 논리적 모순이 없음.
 */
interface Shape {
    int getArea();
}

class Rectangle implements Shape {
    private int width;
    private int height;

    public Rectangle(int width, int height) {
        this.width = width;
        this.height = height;
    }

    public int getArea() { return width * height; }
}

class Square implements Shape {
    private int side;

    public Square(int side) {
        this.side = side;
    }

    public int getArea() { return side * side; }
}
```

```JAVA
/**
 * [Bad Case] ISP 위반
 * 복합기 인터페이스 하나에 모든 기능을 넣음.
 * 단순 프린터기는 필요 없는 scan(), fax()까지 억지로 구현해야 함.
 */
interface SmartDevice {
    void print();
    void scan();
    void fax();
}

class SimplePrinter implements SmartDevice {
    public void print() { System.out.println("출력합니다."); }
    
    // 사용하지 않는 기능을 빈 통으로 두거나 예외를 던져야 함
    public void scan() { /* 미지원 */ }
    public void fax() { /* 미지원 */ }
}

/**
 * [Good Case] ISP 준수
 * 인터페이스를 기능별로 쪼갬.
 * 각 기기는 자신에게 필요한 인터페이스만 골라서 구현하면 됨.
 */
interface Printer {
    void print();
}

interface Scanner {
    void scan();
}

interface Fax {
    void fax();
}

// 일반 프린터는 출력 기능만 가짐
class BasicPrinter implements Printer {
    public void print() { System.out.println("출력합니다."); }
}

// 복합기는 여러 인터페이스를 다중 구현
class MultiFunctionMachine implements Printer, Scanner, Fax {
    public void print() { System.out.println("출력합니다."); }
    public void scan() { System.out.println("스캔합니다."); }
    public void fax() { System.out.println("팩스를 보냅니다."); }
}
```

```JAVA
/**
 * [Bad Case] DIP 위반
 * Switch가 특정 전구인 'LightBulb' 클래스를 직접 알고(의존) 있음.
 * 만약 LED 전구로 바꾸고 싶다면 Switch 클래스의 코드도 수정해야 함. (결합도가 높음)
 */
class LightBulb {
    void turnOn() { System.out.println("전구가 켜졌습니다."); }
}

class Switch {
    private LightBulb bulb; // 구체적인 클래스에 직접 의존

    Switch() { this.bulb = new LightBulb(); }

    void operate() {
        bulb.turnOn();
    }
}

/**
 * [Good Case] DIP 준수
 * Switch와 전구 모두 'Switchable'이라는 인터페이스(추상화)에 의존함.
 * 이제 어떤 전구(LED, 형광등)가 오더라도 Switch 코드는 바뀔 필요가 없음.
 */
interface Switchable {
    void turnOn();
}

class LightBulb implements Switchable {
    @Override
    public void turnOn() { System.out.println("전구가 켜졌습니다."); }
}

class LEDLamp implements Switchable {
    @Override
    public void turnOn() { System.out.println("LED 램프가 밝게 켜졌습니다."); }
}

class Switch {
    private Switchable device; // 인터페이스에 의존

    // 외부에서 실제 객체를 주입받음 (의존성 주입)
    Switch(Switchable device) {
        this.device = device;
    }

    void operate() {
        device.turnOn();
    }
}
```
