[컴퓨터공학과 학부생이 졸업전에 쓰는 AI에 대한 고찰 ]

### AI는 휼륭한 설계자가 될 수 없다. 

우리는 프로젝트를 단리가 아닌 복리로 봐야한다. 

이전에 작성한 프로그램에서 모듈을 추가하고 기존 기능을 유지보수를 함으로서 지금 만드는 것이 다음 것을 위한 자원이 된다.  

현재의 AI는 단일 기능을 구현하는데에는 좋을지 몰라도, 전체적인 프로젝트 상황을 보고 설계를 진행하는 시니어 엔지니어의 요령을 따라하지 못한다. 


개발자들은 ai를 더 잘 쓰기 위해 agent.md파일을 작성하고 어떻게 작성해야 더 효율적인 결과를 뽑을 수 있는지에 대한 방법을 공유한다. 

하지만 더 나은 agent.md파일로는 "복리게임"을 이길 수 없다. 

AI가 제시하는 코드는 그럴듯하고 훌륭하게 작동하지만, 거기에는 우리 프로젝트만의 고유한 '맥락(Context)'과 **미래에 대한 고민**이 빠져있다. 

AI가 내놓은 정답은 방대한 데이터의 훌륭한 평균치일 뿐, 내 서비스의 아키텍처를 책임져주지 않는다.

**단리적인 성과(빠른 기능 구현)에 취해 비판적 사고를 멈추는 순간, 우리의 프로젝트는 유지보수가 불가능한 누더기 코드가 되어버린다.**

"이 코드가 정말 최선인가?", "이 구조가 우리 시스템에 맞는가?"를 의심하고 타당성을 검토하는 것. 

파편화된 조각들을 모아 전체 시스템의 그림을 그리는 것이야말로 복리 게임을 이끌어가는 진짜 개발자의 역할이다.


그래서 우리는 질문을 바꿔야 한다. 

'어떻게 프롬프트를 잘 쓸까?'가 아니라, 'AI와 공존하기 위해 **우리는 어떤 사고방식과 마음가짐을 가져야 할까?**' 

이 글은 그 질문에 대한 나의 답이다.

---

### 유지보수를 전혀 신경쓰지않는 제미나이?

우리 학교에는 오픈소스를 활용해 팀만의 아이디어를 웹이나 앱으로 구현하는 "공개 소프트웨어 프로젝트"라는 과목이 있다.

나는 이 수업에서 최소 1인분을 하기 위해서 겨울방학 시즌에 동아리에서 벡엔드 스터디를 진행하였다. 

난 Spring Boot3을 빠르게 학습하고 DB연동을 해서 아주 간단한 블로그를 만드는 것까지 공부했다. 

덕분에 기능 하나를 만들기 위해 Controller, Service, Repository 레이어를 철저히 분리해야 하는 **계층형 아키텍처**의 기본기를 다질 수 있었다.


본격적으로 공개 소프트웨어 프로젝트에서 아이디어를 만들고 구현을 시작할 즈음이었다. 

Spring Boot3은 많은 사람들이 사용하는 프레임워크다보니 개발자의 편의를 돕는 기능 또한 많이 제공한다.

그 중 하나가 어노테이션(Annotaoin)이다. 

단 한 줄의 어노테이션만 붙이면 프레임워크가 알아서 반복적인 보일러플레이트 코드를 생성해 주는데, 제미나이는 이를 완전히 무시한 채 원시적인 코드를 일일이 길게 늘어쓰고 있었던 것이다.

나야 미리 공부를 해온 상태라 "얘가 왜이러지?" 하고 문제점을 인식했지만 다른 팀원들은 Spring 사용경험이 전혀 없어 어노테이션의 존재를 전혀 모르던 참이라 이 코드가 비효율적이라는 것을 전혀 모르고 있었다. 

그 외에 controller와 Service 레이어를 명확히 나누어 책임을 분리해야 할 코드들을, 제미나이는 파일 하나에 통째로 때려 넣는 이른바 '스파게티 코드'를 자랑스럽게 내밀었다

전체 그림을 전혀 보지 못하고 내가 준 프롬프트에 대한 코드만을 제공해주기만 했다. 

나는 이때부터 제미나이에 대한 의존도를 낮출 필요성을 느끼기 시작했다. 

당시 상황 예시 코드
```java
// 내가 원한 코드 
import lombok;

@Getter
@Setter
public class User{
    private Long id;
    private String name;
    private String email;
}


// 제미나이가 준 코드
public class User {
    private Long id;
    private String name;
    private String email;

    public Long getId() {
        return id;
    }

    public String getName() {
        return name;
    }

    public String getEmail() {
        return email;
    }
    public void setId(Long id) {
        this.id = id;
    }

    public void setName(String name) {
        this.name = name;
    }

    public void setEmail(String email) {
        this.email = email;
    }
}
// 제미나이가 준 코드처럼 작성하게 되면 엔티티에 속성이 추가될 때 마다 get, set 코드를 매번 추가해야하는 번거로움이 있다. 
// 깜박하고 작성 안하는 순간 코드 에러가...
```


**얘는 모듈화에 완전 취약한 툴이다!!**

AI에 의존하기만 하면 무조건 망한다는걸 직감했다. 

### 비판적 사고를 해라!

3학년을 끝마치고 겨울방학때 OS랩 학부인턴을 들어가게 되었다. 

나는 그곳에서 **생각하는 법**에 대해 배웠다. 

연구자는 비판적으로 생각할 수 있어야 한다. 

수많은 논문을 읽으면서 단순히 내용을 수용하는 데 그치면 안 된다. 

"이 논문에서 내가 얻어갈 수 있는 핵심 아이디어가 뭐지?", "저자의 이 설계가 과연 항상 맞을까?"라며 끊임없이 묻고 의심해야 한다.

생각하는 것을 멈추면 안된다. 

생각을 이어가다보면 이러한 질문에 이끌리게 된다. 

그럼 AI시대에 우리 개발자들의 역할은 무엇일까? 

나는 이 생각하는 방법을 AI에게 적용하기 시작했다. 

사실 그전까지만 해도 나는 제미나이나 GPT 같은 AI를 그저 빠르고 똑똑한 '고급 검색 툴' 정도로만 사용하고 있었다. 

AI가 정답처럼 보이는 코드를 짜주면 별다른 의심 없이 가져다 쓰기 바빴다. 

앞선 백엔드 프로젝트에서 AI의 얄팍함을 엿보긴 했지만, 여전히 도구에 끌려다니고 있었던 것이다.

하지만 연구실에서 교수님께 1대1로 비판적 사고를 훈련받으며, 나는 AI가 던져주는 결과물을 그대로 받아들이는 것이 아니라, 의심하고 과연 이게 정답일까 에 대해 집요하게 생각하기 시작했다. 

물론 아직 마음처럼 노련한 사고를 하지는 못한다. 

### 질문이 달라져야 대답을 의심할 수 있다. 

**우리에게 필요한 사고**
"이게 정말 맞을까?" 라고 의심하는 **비판적 사고**가 필요하다. 

비판적 사고를 제대로 작동시키려면, AI가 던져주는 파편들을 검증하기 이전에 우리가 어떤 아키텍처를 원하고 어떤 제약사항이 있는지를 먼저 명확히 설명해야한다. 

그동안 나는 AI를 궁예처럼 취급하여 디테일한 설명을 주지 않고 그저 단편적인 지시, 혹은 검색 요청만을 내리고 있었다.

하지만 질문이 빈약하면 답변도 빈약한 법. 

나는 이 생각하는 방법을 AI와 융합시키기 위해 먼저 프롬프트 작성법에 대한 공부를 시작하였다. 

---

**핵심원칙**
- AI에게 정답을 요구하지말고 대화하라
- 더 나아가 AI에게 질문하지말고 AI가 나에게 질문하게 하라.
- AI는 검색엔진이 아니다. 대화상대, co-worker로 취급해주자. 
  - 피드백을 주고 코칭하고 질문을 받아주자. 
- 너가 맥락을 제공안하면 AI는 너의 상황을 고려하지 않고 general 하게 코드를 생성해준다. 
  - 나의 경우에는 real time system에 사용할 code를 요청했더니 코드 사방에 lock를 걸어두었다. 
  - 따로 명시적으로 lock-free 코드를 작성해달라고 해야한다. 

**Harness Engineering 은 어떻게 작성하는것인가.**

연구와 설계에서는 **정보의 누락**이 없어야한다. 

- Objective : 해결하고자 하는 구체적인 목표
- Contraints : 반드시 지켜야 할 기술적 제약 (ex. python 3.11 사용, NoSQL 금지)
- Input data : 참고해야할 기존 코드나 논문 초록
- Output style : 결과물의 형태


**1. 멀티 에이전트 비판 하니스 (Self-Criticism Loop)**

AI 내부에서 '개발자'와 '리뷰어'의 역할을 나누어 스스로 토론하게 만드는 고급 기법

- 제약 조건: 최적화 방안을 제시할 때, 먼저 장점을 설명하고 그 다음으로 발생할 수 있는 부작용(예: CPU 점유율 상승 등)을 최소 2가지 이상 명시해.
- 제약 조건: 네가 제안한 방법보다 더 나은 대안이 있는지 스스로 비판하고 최종안을 확정해.
- 목적: 한쪽으로 치우친 최적화가 아닌, 시스템 전체의 균형을 고려한 제안을 받게 함


**2. 환경 전제 조건 하니스 (Pre-condition Harness)**

질문할 때마다 배경을 설명하지 않아도 되도록 **기본 설정값을 고정**하는 장치입니다.

- 제약 조건: 별도의 언급이 없으면 RMW는 Fast DDS를 기준으로 하고, OS는 Ubuntu 22.04 LTS 환경으로 가정해.
- 제약 조건: 실시간성(Real-time)이 중요한 환경임을 항상 전제로 답변해.
- 목적: 매번 상황을 설명하는 컨텍스트 주입 과정을 생략하고 핵심에 집중하게 함

**3. 질문을 주고받는 상호작용 하니스 (Coaching Loop)**

단순히 답만 주는 게 아니라, 제가 사용자님에게 다시 질문을 던지게 만드는 설정입니다.

- 설정: 답변 마지막에는 항상 내가 놓쳤을 법한 변수나, 사용자님의 의도를 묻는 질문을 하나씩 던져줘.
- 효과: 사용자님이 미처 생각하지 못한 엣지 케이스(Edge Case)를 스스로 떠올리게 유도하여 **사고의 폭을 넓혀줍니다.**

**4. 논리적 근거 요구 하니스 (Chain of Thought)**

결과만 보고 판단하는 게 아니라 과정에 집중하는 설정입니다.

- 설정: 최적화 방안을 제시할 때, 왜 그 수치가 나왔는지 논리적 근거를 먼저 설명하고 결론을 내줘.
- 효과: 제가 생각하는 논리 구조를 사용자님이 비판적으로 검토하면서, 자연스럽게 기술적 통찰력을 공유하게 됨

**5. 실패 사례 및 부작용 경고 하니스 (Red Teaming)**

긍정적인 결과만 기대하는 게 아니라 위험 요소를 관리하는 설정입니다.

- 설정: 내가 제안한 방식이 실전에서 실패할 수 있는 시나리오를 반드시 하나 포함해줘.
- 효과: 실무에서 발생할 수 있는 리스크 관리 능력을 함께 키울 수 있음

---

AI는 거울이다. 

당신이 AI에게 맥락과 요구사항을 제대로 비춰주지 않으면 AI는 그걸 전혀 비춰주지 못한다. 

무비판적인 수용을 멈추고 질문의 수준을 높이자.

이제 단리적인 '빠른 코딩'을 넘어, 내일을 고민하는 엔지니어로서의 진짜 **'복리 게임'**을 시작할 때다.
