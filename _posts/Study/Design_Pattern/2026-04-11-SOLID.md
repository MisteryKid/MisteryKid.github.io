# SOLID 원칙 핵심 요약 및 예제

### 1. SRP (단일 책임 원칙)
* **개념:** 클래스는 단 하나의 책임(변경될 이유)만 가져야 함.
* **실생활 예제:** 빵집에서 제빵사(BreadBaker)는 빵만 구워야 함. 재고 관리, 청소, 고객 응대 등을 혼자 다 하면 안 됨. 각각의 역할을 분리해야 함.

### 2. OCP (개방/폐쇄 원칙)
* **개념:** 확장에는 열려(Open) 있고, 수정에는 닫혀(Closed) 있어야 함.
* **실생활 예제:** 결제 시스템에 'PayPal 결제'를 추가할 때, 기존의 '신용카드 결제' 코드를 뜯어고치는 것이 아니라 새로운 PayPal 클래스를 추가해서 확장함.

### 3. LSP (리스코프 치환 원칙)
* **개념:** 자식 클래스는 부모 클래스의 역할을 완벽히 대체할 수 있어야 함. (부모의 규칙을 깨면 안 됨)
* **실생활 예제:** 직사각형(부모)을 상속받아 정사각형(자식)을 만들면 안 됨. 직사각형은 가로/세로를 따로 변경할 수 있지만, 정사각형은 같이 변경되어야 하므로 부모의 동작 방식을 위반하게 됨.

### 4. ISP (인터페이스 분리 원칙)
* **개념:** 사용하지 않는 기능(메서드)을 강제로 구현하게 만들면 안 됨. 인터페이스는 작고 구체적으로 쪼개야 함.
* **실생활 예제:** 단순 프린터 기기에게 '스캔', '팩스' 기능까지 포함된 거대한 복합기 인터페이스를 상속받게 하면 안 됨.

### 5. DIP (의존성 역전 원칙)
* **개념:** 고수준 모듈(메인 로직)이 저수준 모듈(세부 구현)에 직접 의존해선 안 되며, 둘 다 '추상화(인터페이스)'에 의존해야 함.
* **실생활 예제:** 전등 스위치가 특정 전구 브랜드를 직접 알고(결합) 있으면 안 됨. 어떤 전구가 오든 켜고 끌 수 있는 '스위치 규격'을 만들어야 함.


## SOLID를 준수해서 프로그래밍하면 생기는 장점

네이버 페이 결제 기능을 담당하는 코드를 작성했다고 가정해보자. 

```java
// SOLID를 무시한 복합 결제 매니저 클래스
public class PaymentManager {
    
    // 네이버페이 전용 결제 메서드
    public void processNaverPay(int amount) {
        System.out.println("네이버페이 서버 연결...");
        System.out.println(amount + "원 결제 완료");
        
        // 🚨 영수증 출력 로직까지 한 클래스에 다 들어있음
        System.out.println("종이 영수증 발행 중..."); 
    }
}

```

그런데 나중에 아래의 요청이 추가적으로 온다고 생각해보자. 
1. 카카오페이 결제 기능도 추가해 주세요!"
2. "환경 보호를 위해 종이 영수증 말고 모바일 알림톡 영수증으로 바꿔주세요!"

SOLID를 안 지키면 생기는 일
- 개발자는 PaymentManager 클래스의 내부 코드를 직접 뜯어고쳐야함
- processKakaoPay 메서드를 새로 추가하고, 기존 processNaverPay 안에 있던 영수증 로직을 지우고 알림톡 코드를 넣어야함
- 문제점
  - 네이버페이 코드를 고치다가 아무 상관 없는 기존 기능에 버그가 발생할 수 있다.
  - 클래스 하나가 너무 많은 책임(결제+영수증)을 지고 있고, 확장에 열려있지 않기 때문.

SOLID를 준수했을 때의 장점
- SRP 준수: 결제 클래스와 영수증 클래스를 분리
- OCP 준수: 결제 방식을 인터페이스(Payment)로 추상화

```java
// 1. [OCP] 결제의 공통 역할을 인터페이스로 정의
public interface Payment {
    void pay(int amount);
}

// 2. [SRP] 네이버페이는 오직 '네이버페이 결제'만 책임짐
public class NaverPay implements Payment {
    @Override
    public void pay(int amount) {
        System.out.println("네이버페이 " + amount + "원 결제 완료");
    }
}

// 3. [SRP] 영수증 처리는 별도 클래스로 완전히 분리
public class ReceiptPrinter {
    public void print() {
        System.out.println("모바일 알림톡 영수증 발행");
    }
}
```
똑같은 요구사항 변경이 들어왔을 때

**"카카오페이 추가해 주세요!"**
- 기존 코드는 단 한 줄도 건드리지 않고, 그냥 class KakaoPay implements Payment라는 새로운 파일(클래스) 하나만 새로 만들면 끝
- 장점: 기존에 잘 돌아가던 네이버페이 코드를 건드릴 일이 없으니 새 기능 때문에 기존 기능이 고장 나는 배달 사고(버그)가 0%가 됨

**"종이 영수증 대신 모바일 영수증으로 바꿔주세요!"**
- 결제 코드는 구경도 안 해도 됨. 오직 영수증을 담당하는 ReceiptPrinter 내부만 싹 고치면 됨
- 장점: 코드가 격리되어 있어 수정해야 할 위치를 명확히 알 수 있고, 수정 범위가 최소화

**장점 정리**
- 유지보수성 (Maintainability): 코드가 기능별로 이쁘게 쪼개져 있어서(SRP), 문제가 터졌을 때 어떤 파일의 코드를 고쳐야 하는지 한눈에 보임.

- 확장성 (Extensibility): 새로운 기능을 추가할 때 기존에 완성된 코드를 수정하는 게 아니라, 새로운 클래스를 '조립'하기만 하면 됨(OCP).

- 테스트 용이성 (Testability): 클래스들이 인터페이스를 통해 느슨하게 연결되어 있으므로, 가짜 객체(Mock)를 만들어 특정 기능만 독립적으로 테스트하기 매우 쉬워짐


## LSP 문제 (↓) 
한 줄 요약: "자식 클래스는 언제나 부모 클래스를 대체할 수 있어야 한다." 즉, 부모의 규칙을 자식이 속이면 안 된다는 원칙입니다. (상속 관계에서 하향[↓]으로 내려갈 때 발생)

대표 위반 사례
```java
// 부모 클래스: 직사각형
class Rectangle {
    protected int width;
    protected int height;

    public void setWidth(int width) { this.width = width; }
    public void setHeight(int height) { this.height = height; }
    public int getArea() { return width * height; }
}

// 자식 클래스: 정사각형 (직사각형을 상속받음 ↓)
class Square extends Rectangle {
    // 🚨 정사각형은 가로/세로가 같아야 하므로 부모의 메서드를 왜곡함
    @Override
    public void setWidth(int width) {
        this.width = width;
        this.height = width; // 가로를 바꾸면 세로도 강제로 바꿈!
    }

    @Override
    public void setHeight(int height) {
        this.width = height;
        this.height = height; // 세로를 바꾸면 가로도 강제로 바꿈!
    }
}

```
해결책: 정사각형과 직사각형의 상속 관계를 끊고, 공통 인터페이스(도형)로 묶어야 합니다.


### DIP 문제 (↑) 
한 줄 요약: "구체적인 것에 의존하지 말고, 추상적인 것(인터페이스)에 의존하라." 고수준 모듈이 저수준 모듈에 휘둘리지 않도록 의존 관계를 위[↑]로 역전시키는 것

대표 위반 사례
```java
// 저수준 모듈 (구체적인 타이어)
class SnowTire {
    public void roll() { System.out.println("눈길에서 굴러갑니다."); }
}

// 고수준 모듈 (자동차 본체)
class Car {
    private SnowTire tire; // 🚨 구체적인 클래스를 직접 의존함! (DIP 위반)

    public Car() {
        this.tire = new SnowTire();
    }

    public void drive() { tire.roll(); }
}

```

이걸 다음과 같이 역전시켜주자. 

```java
// 1. 추상적인 인터페이스를 상위에 둡니다.
interface Tire { void roll(); }

// 2. 저수준 타이어들은 이 인터페이스를 구현합니다. (의존성이 위로 향함 ↑)
class SnowTire implements Tire { public void roll() { ... } }
class NormalTire implements Tire { public void roll() { ... } }

// 3. 고수준 자동차는 구체적인 타이어가 아닌 '인터페이스'만 의존합니다.
class Car {
    private Tire tire; // ✨ 추상화에 의존!

    // 외부에서 타이어를 주입(DI)받음
    public Car(Tire tire) { this.tire = tire; } 

    public void drive() { tire.roll(); }
}

```

```JAVA
/**
 * [Bad Case] OCP 위반
 * 새로운 결제 방식(NaverPay)이 추가될 때마다 
 * processPayment 내부의 if-else 문을 계속 수정해야 함. (수정에 열려 있음)
 */
class PaymentService {
    void processPayment(String type) {
        if (type.equals("CreditCard")) {
            System.out.println("신용카드 결제 처리");
        } else if (type.equals("PayPal")) {
            System.out.println("페이팔 결제 처리");
        }
        // 네이버페이가 추가된다면? 또 이 코드를 고쳐야 함.
    }
}

/**
 * [Good Case] OCP 준수
 * 인터페이스를 활용하여 확장은 열려 있고, 
 * 기존 PaymentService의 코드는 수정할 필요가 없음. (수정에 닫혀 있음)
 */
interface Payment {
    void process();
}

class CreditCardPayment implements Payment {
    public void process() { System.out.println("신용카드 결제 처리"); }
}

class PayPalPayment implements Payment {
    public void process() { System.out.println("페이팔 결제 처리"); }
}

// 새로운 결제 수단 추가 시 기존 코드 수정 없이 클래스만 생성하면 됨
class NaverPayPayment implements Payment {
    public void process() { System.out.println("네이버페이 결제 처리"); }
}

class PaymentService {
    void processPayment(Payment payment) {
        payment.process(); // 어떤 결제가 들어오든 이 로직은 변하지 않음
    }
```

```JAVA
/**
 * [Bad Case] LSP 위반
 * 정사각형은 직사각형의 일종이라 생각해서 상속받았으나,
 * 부모의 메서드(가로/세로 독립 설정)를 마음대로 바꿔버려 부모의 기대를 저버림.
 */
class Rectangle {
    protected int width;
    protected int height;

    public void setWidth(int width) { this.width = width; }
    public void setHeight(int height) { this.height = height; }
    public int getArea() { return width * height; }
}

class Square extends Rectangle {
    @Override
    public void setWidth(int width) {
        this.width = width;
        this.height = width; // 정사각형이라서 세로까지 강제로 바꿈
    }

    @Override
    public void setHeight(int height) {
        this.width = height; // 정사각형이라서 가로까지 강제로 바꿈
        this.height = height;
    }
}
// 결과적으로 Rectangle 타입으로 Square를 쓸 때 가로만 바꿨는데 
// 세로까지 바뀌어버리면 Rectangle을 사용하는 사용자 입장에서 당황스러운 상황이 됨.


/**
 * [Good Case] LSP 준수
 * 두 도형의 공통점(넓이 구하기)만 추출하여 인터페이스로 정의함.
 * 이제 부모(Shape)의 역할을 자식이 수행할 때 논리적 모순이 없음.
 */
interface Shape {
    int getArea();
}

class Rectangle implements Shape {
    private int width;
    private int height;

    public Rectangle(int width, int height) {
        this.width = width;
        this.height = height;
    }

    public int getArea() { return width * height; }
}

class Square implements Shape {
    private int side;

    public Square(int side) {
        this.side = side;
    }

    public int getArea() { return side * side; }
}
```

```JAVA
/**
 * [Bad Case] ISP 위반
 * 복합기 인터페이스 하나에 모든 기능을 넣음.
 * 단순 프린터기는 필요 없는 scan(), fax()까지 억지로 구현해야 함.

interface SmartDevice {

class SimplePrinter implements SmartDevice {

 * [Good Case] ISP 준수

interface Printer {
    void print();
}

interface Scanner {
    void scan();
}

interface Fax {
    void fax();
}

// 일반 프린터는 출력 기능만 가짐
class BasicPrinter implements Printer {

// 복합기는 여러 인터페이스를 다중 구현
class MultiFunctionMachine implements Printer, Scanner, Fax {
```

```JAVA
/**
 * [Bad Case] DIP 위반
 * Switch가 특정 전구인 'LightBulb' 클래스를 직접 알고(의존) 있음.
 * 만약 LED 전구로 바꾸고 싶다면 Switch 클래스의 코드도 수정해야 함. (결합도가 높음)
 */
class LightBulb {
    void turnOn() { System.out.println("전구가 켜졌습니다."); }
}

class Switch {
    private LightBulb bulb; // 구체적인 클래스에 직접 의존

    Switch() { this.bulb = new LightBulb(); }

    void operate() {
        bulb.turnOn();
    }
}

/**
 * [Good Case] DIP 준수
 * Switch와 전구 모두 'Switchable'이라는 인터페이스(추상화)에 의존함.
 * 이제 어떤 전구(LED, 형광등)가 오더라도 Switch 코드는 바뀔 필요가 없음.
 */
interface Switchable {
    void turnOn();
}

class LightBulb implements Switchable {
    @Override
    public void turnOn() { System.out.println("전구가 켜졌습니다."); }
}

class LEDLamp implements Switchable {
    @Override
    public void turnOn() { System.out.println("LED 램프가 밝게 켜졌습니다."); }
}

class Switch {
    private Switchable device; // 인터페이스에 의존

    // 외부에서 실제 객체를 주입받음 (의존성 주입)
    Switch(Switchable device) {
        this.device = device;
    }

    void operate() {
        device.turnOn();
    }
}
```
