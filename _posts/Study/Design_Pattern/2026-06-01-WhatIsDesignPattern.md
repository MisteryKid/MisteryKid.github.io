## GoF (Gang of Four)
https://www.geeksforgeeks.org/system-design/gang-of-four-gof-design-patterns/

수학, 물리학과 달리 컴퓨터 공학은 역사가 무척이나 짧다. 

역사가 짧았기에 코드를 어떻게 더 깔끔하고 재사용하기 좋게 작성할 수 있는 지에 대한 명확한 가이드라인이 없을 때였다. 

이러한 혼란 속에서 개발자들은 "처음부터 완벽하게 설계할 수 없다면, 이미 검증된 우수한 구조를 공식처럼 만들어 공유하면 어떨까?"라는 고민을 하기 시작하였다. 

그 결과물이 바로 GoF의 디자인 패턴이다. 

GoF는 건축 공학에서 사용하던 디자인 패턴의 개념을 소프트웨어 공학 분야에서 처음으로 명확히 정리한 4명의 사람들이다. 


- **에리히 감마 (Erich Gamma)**: Eclipse(이클립스) 개발을 이끌었고, 현재 마이크로소프트에서 VS Code(Visual Studio Code) 개발에 참여한 거물

- **리처드 헬름 (Richard Helm)**: IBM T.J. 왓슨 연구소에서 객체지향 기술을 연구한 핵심 연구원

- **랄프 존슨 (Ralph Johnson)**: 일리노이 대학교 교수로, 객체지향 및 리팩토링 분야의 세계적인 권위자

- **존 블라시디스 (John Vlissides)**: 스탠퍼드 대학 출신으로 IBM 연구원으로 활동했으며, 디자인 패턴의 대중화에 큰 기여

이렇게 23가지의 디자인 패턴을 제안하면서 설계 및 개발에서 흔히 접하는 문제에 대한 해결 가이드 라인을 제공할 수 있었다. 

이들은 두가지 원칙을 내세웠다. 

구현이 아닌 인터페이스에 맞춰 프로그래밍 하라
- 구체적인 구현체를 직접 의존하면 나중에 코드를 바꿀 때 다 깨질 수 있다. 대신 인터페이스를 의존하게 만들어 유연성을 높이자. 


상속 보다는 합성을 선호해라. 
- 상속은 부모와 자식 클래스가 너무 강하게 결합되어 유연성이 떨어진다. 그보다 다른 객체를 내부에 포함(합성)하여 필요할 때 조립해 쓰자.

## 상속의 문제점과 위임

상속은 부모의 내부 구현이 자식에게 그대로 노출된다는 치명적인 약점이 있다. 이를 객체지향에서는 캡슐화가 깨진다고 표현한다. 

그럼 어떨 때 문제가 발생할까?


1. 스택을 vector로 구현했을 떄. 

```java
// Java 실제 내부 구조 (설계 오류의 예)
public class Stack<E> extends Vector<E> {
    public E push(E item) {
        addElement(item);
        return item;
    }

    public synchronized E pop() {
        // 마지막 원소를 꺼내는 로직
    }
}

```
Stack이 Vector를 상속받는 순간, Vector가 가진 모든 메서드가 Stack 외부로 노출된다. 

Vector에는 아무 위치에나 값을 넣을 수 있는 insertElementAt() 같은 메서드가 있는데 얘도 노출된다.

따라서 이론상 아래와 같은 코드 가능해짐
```java
Stack<String> stack = new Stack<>();
stack.push("첫 번째");
stack.push("두 번째");

// 💥 문제 발생: 스택인데 부모(Vector) 메서드를 마음대로 호출할 수 있음!
stack.insertElementAt("중간에 끼어들기", 1); 
stack.removeElementAt(0); // 첫 번째 데이터를 강제로 삭제함
```

위임으로 설계하면 다음과 같이 코드를 작성할 수 있다. 
```java
// 위임을 사용한 안전한 스택 설계
public class BetterStack<E> {
    
    // 1. {중요!!!!} Vector를 상속받지 않고, 내부 부품(필드)으로 숨깁니다. (캡슐화)
    private Vector<E> vector = new Vector<>();

    // 2. 외부에는 오직 '스택' 본연의 기능만 공개합니다.
    public E push(E item) {
        vector.addElement(item); // 내부 부품에 일을 맡김 (위임)
        return item;
    }

    public E pop() {
        if (vector.isEmpty()) throw new EmptyStackException();
        E obj = vector.lastElement();
        vector.removeElementAt(vector.size() - 1); // 위임
        return obj;
    }
    
    // 💡 Vector의 insertElementAt 같은 메서드는 외부에 노출되지 않음!
    // 따라서 메인문에서 BetterStack 객체를 만들면 insertElemtnat 호출 불가함
}
```

정리하면 main문에서 BetterStack를 호출하면 여기 메서드에서 정의한거만 쓸 수 있는데 만약 상속을 받으면 다른 벡터 메소드도 호출이 가능해짐

그러면 의도와 다른 동작이 가능해지므로 되도록이면 has-a 관계인 합성을 사용하도록 하고 있다.

디자인 패턴은 크게 3가지로 나눌 수 있다. 

### 생성 패턴
- 소프트웨어 개발에서 **객체 생성 과정에 초점**을 맞춘다. 
- 후에 우리가 조각들을 어떻게 조립하는지에 대한 복잡한 세부 사항을 숨긴다. 

### 구조 패턴
- 객체를 결합하여 더 큰 구조를 만드는데 필요한 지침을 제공한다. 
- 이러한 구조를 유연하고 재사용가능하며 유지 관리가 용이하도록 만드는제 중점을 둔다. 
- 객체 구성 및 상호 작용을 관리하는 명확한 방법을 제공하여 시스템 설계를 간소화한다. 

### 행위 패턴
- 객체들이 서로 상호작용하고 소통하는 방식에 초점을 맞춘다. 
- 기존 기능을 손상시키지 않고 소프트웨어를 더 쉽게 수정, 유지 관리 및 확장할 수 있도록 한다. 


## 생성패턴

### 싱글톤 패턴
- **개념:** 클래스의 인스턴스가 오직 하나만 생성되도록 보장하고, 이 인스턴스에 접근할 수 있는 전역적인 접점을 제공함.
- **용도:** 스레드 풀, 캐시, 로그 기록, 설정 관리 등 전역에서 하나의 객체만 공유해야 할 때 사용함.
- **예시:** 애플리케이션 전체의 로그를 파일에 기록하는 단 하나의 `Logger` 객체, 혹은 데이터베이스 연결을 관리하는 `ConnectionPool` 인스턴스임.

### 팩토리 메서드 패턴
- **개념:** 객체 생성 인터페이스를 정의하되, 어떤 클래스의 인스턴스를 생성할지는 서브클래스에서 결정하도록 팩토리 메서드로 위임하는 패턴임.
- **용도:** 클래스가 자신이 생성해야 하는 구체적인 객체의 종류를 예측할 수 없거나, 객체 생성 로직과 실제 사용 로직을 분리하여 결합도를 낮추고 싶을 때 사용함.
- **예시:** 카드 결제 시스템에서 `PaymentCreator`라는 상위 클래스를 두고, 하위의 `NaverPaymentCreator`는 `NaverCard` 객체를, `KakaoPaymentCreator`는 `KakaoCard` 객체를 생성하여 반환하도록 만드는 구조임.

### 추상 팩토리 패턴
- **개념:** 구체적인 클래스를 지정하지 않고, 연관되거나 의존적인 객체들의 제품군(패밀리)을 한꺼번에 생성할 수 있는 인터페이스를 제공하는 패턴임.
- **용도:** 시스템이 여러 제품군 중 하나로 구성되어야 하고, 생성되는 관련 객체들이 서로 항상 일관되게 짝을 맞춰 작동해야 할 때(예: OS별 UI 컴포넌트 묶음) 사용함.
- **예시:** OS 테마 시스템에서 `WindowsFactory`는 윈도우 스타일의 버튼과 텍스트 박스를 세트로 생성하고, `MacFactory`는 맥 스타일의 버튼과 텍스트 박스를 세트로 생성하여 UI의 일관성을 보장하는 구조임.

### 빌더 패턴
- **개념:** 복잡한 객체의 생성 과정과 표현 방법을 분리하여, 동일한 생성 절차(인터페이스)에서도 서로 다른 표현 결과를 만들 수 있게 하는 패턴임.
- **용도:** 생성자에 들어가는 매개변수가 너무 많아 인자의 순서가 헷갈리거나(점진적 생성자 문제), 객체를 단계별로 동적으로 조립하여 생성해야 할 때 사용함.
- **예시:** 수십 개의 옵션이 존재하는 수제 햄버거 주문 시스템에서 `Hamburger.Builder`를 사용하여 빵 종류, 패티 종류, 치즈 추가 여부 등을 메서드 체이닝 방식으로 명확하게 조합하여 최종 `Hamburger` 객체를 완성하는 구조임.

### 프로토타입 패턴
- **개념:** 매번 새 인스턴스를 처음부터 생성하지 않고, 원본 객체의 인스턴스를 복사(Clone)하여 새로운 객체를 생성하는 패턴임.
- **용도:** 객체를 새로 생성하는 비용(데이터베이스 조회, 복잡한 초기화 연산 등)이 너무 커서 기존 데이터를 복사해 쓰는 것이 훨씬 효율적일 때 사용함.
- **예시:** DB에서 수만 건의 회원 정보를 조회하여 무겁게 초기화된 통계용 기본 리포트 객체가 있을 때, 다음 통계 연산을 위해 매번 DB를 새로 조회하지 않고 기존 리포트 객체를 `clone()`으로 복제하여 값만 조금씩 수정해 사용하는 방식임.

## 구조 패턴

### 어댑터 패턴 (Adapter)
- **개념(정의):** 호환성이 없는 인터페이스를 가진 두 클래스가 함께 작동할 수 있도록 중간에서 변환 역할을 해주는 클래스를 두는 패턴임. 상속을 이용하는 클래스 어댑터와 위임(합성)을 이용하는 객체 어댑터 방식으로 나뉘며, 일반적으로 위임 방식을 선호함.
- **목적과 필요성:** 이미 잘 만들어져 검증된 기존 클래스나 외부 라이브러리가 존재하지만, 현재 시스템이 요구하는 인터페이스와 맞지 않을 때 기존 코드를 수정하지 않고 그대로 재사용하기 위함임.
- **예시:** 110V 가전제품(외부 라이브러리)을 국내의 220V 콘센트(현재 시스템)에 꽂아 쓰기 위해 중간에 돼지코 플러그(어댑터)를 연결하여 호환성을 맞추는 구조임.

### 브릿지 패턴 (Bridge)
- **개념(정의):** 기능의 인터페이스 계층과 실제 기능을 구현하는 구현 계층을 완전히 분리하고, 두 계층 사이에 링크(다리) 역할을 하는 멤버 변수(위임 인터페이스)를 두어 상호작용하게 하는 패턴임.
- **목적과 필요성:** 상속을 사용해 기능을 확장하면 클래스 폭발(클래스 개수가 기하급수적으로 늘어남)이 발생하므로, 추상화와 구현을 분리하여 양쪽 계층이 독립적으로 확장·변경될 수 있도록 결합도를 낮추기 위함임.
- **예시:** 여러 종류의 리모컨(기능 계층: 일반 리모컨, 스마트 리모컨)이 내부 필드로 TV 인터페이스(구현 계층: 삼성 TV, LG TV)를 가져, 리모컨 종류와 TV 제조사가 상속으로 얽히지 않고 독립적으로 확장할 수 있게 만드는 구조임.

### 컴포지트 패턴 (Composite)
- **개념(정의):** 객체들을 트리 구조로 구성하여 단일 객체(Leaf)와 복합 객체(Composite)를 동일한 상위 인터페이스로 묶어 다룰 수 있게 하는 패턴임. 클라이언트는 대상이 단일 객체인지 복합 객체 그룹인지 구분하지 않고 동일한 메서드로 접근함.
- **목적과 필요성:** 파일과 폴더 구조, 회사 조직도처럼 전체-부분 관계를 가지는 계층적 객체 구조를 효율적으로 표현하고, 복합적인 객체 그룹을 단일 객체와 똑같은 방식으로 단순하게 제어하기 위함임.
- **예시:** 컴퓨터 파일 시스템에서 단일 파일(Leaf)과 폴더(Composite)가 모두 `Node`라는 공통 인터페이스를 구현하게 하여, 사용자가 파일이든 폴더든 가리지 않고 동일하게 `delete()` 메서드로 삭제할 수 있도록 처리하는 구조임.

### 데코레이터 패턴 (Decorator)
- **개념(정의):** 객체의 핵심 기능은 그대로 두고, 상속 대신 위임(합성) 방식을 사용하여 동적으로 새로운 책임(기능)을 런타임에 겹겹이 추가할 수 있도록 만드는 패턴임. 장식자 클래스 내부에 실제 작동할 대상 객체를 필드로 가짐.
- **목적과 필요성:** 상속을 통한 기능 확장은 컴파일 시점에 고정되어 유연성이 떨어지므로, 객체에 기능을 추가하거나 제거할 때 기존 코드의 수정 없이 유연하게 조합하여 기능을 유동적으로 확장하기 위함임.
- **예시:** 자바의 `BufferedReader br = new BufferedReader(new FileReader("file.txt"))` 구조처럼, 파일 읽기라는 기본 스트림 객체에 버퍼링이라는 장식(기능)을 겉에 씌워 동적으로 성능을 확장하는 구조임.

### 퍼사드 패턴 (Facade)
- **개념(정의):** 복잡하고 거대한 서브시스템 계층(여러 클래스들의 얽히고설킨 호출 구조)에 대해, 가장 앞단에 단순하고 일관된 고수준 인터페이스(통합 창구) 하나를 제공하는 패턴임.
- **목적과 필요성:** 클라이언트가 복잡한 내부 라이브러리나 서브시스템의 복잡한 사용법을 일일이 알 필요 없이, 단 하나의 직관적인 메서드 호출만으로 복잡한 작업을 안전하고 편리하게 처리하게 만들기 위함임.
- **예시:** 영화를 보기 위해 앰프 켜기, 조명 낮추기, 프로젝터 켜기, 스크린 내리기 등의 복잡한 내부 과정을 일일이 조작하지 않고, 스마트 홈의 `HomeTheaterFacade` 클래스가 제공하는 `watchMovie()` 버튼 하나만 눌러 일괄 제어하는 구조임.

### 플라이웨이트 패턴 (Flyweight)
- **개념(정의):** 수많은 유사 객체를 생성해야 할 때, 각 객체마다 데이터를 따로 가지지 않고 가능한 많은 데이터를 공유(Sharing)하여 메모리 사용량을 최적화하는 패턴임. 변하지 않는 속성(본질적 상태)을 공유 풀에 상주시켜 재사용함.
- **목적과 필요성:** 가상 현실 게임의 수많은 나무, 총싸움 게임의 수많은 총알처럼 화면에 같은 종류의 객체가 대량으로 나타날 때, 인스턴스 남발로 인한 메모리 고갈(OOM)을 방지하고 시스템 성능을 보존하기 위함임.
- **예시:** 스타크래프트 게임에서 맵에 등장하는 수천 마리의 '저글링' 객체들이 각각 독립된 그래픽 이미지 데이터를 가지지 않고, 저글링의 공통 외형 이미지는 단 하나의 공유 풀(Flyweight)에서 참조하고 좌표값과 체력만 각자 관리하는 구조임.

### 프록시 패턴 (Proxy)
- **개념(정의):** 실제 작업을 수행하는 대상 객체(Real Subject)에 직접 접근하지 않고, 중간에 대리인 객체(Proxy)를 두어 접근을 제어하고 가로채는 패턴임. 프록시는 실제 객체와 동일한 인터페이스를 구현하며 내부에 실제 객체를 위임 필드로 가짐.
- **목적과 필요성:** 실제 객체를 생성하는 비용이 너무 커서 필요할 때까지 생성을 미루거나(가상 프록시), 객체 접근에 대한 권한 검증이 필요하거나(보호 프록시), 원격 서버의 객체를 로컬처럼 호출해야 할 때(원격 프록시) 본래 비즈니스 로직을 건드리지 않고 제어 흐름을 추가하기 위함임.
- **예시:** 대용량 이미지를 게시판 목록에 띄울 때, 처음부터 무거운 이미지 원본 객체를 다 로딩하지 않고 가짜 이미지 객체(Proxy)를 먼저 띄워 둔 뒤, 사용자가 클릭하는 순간에만 실제 이미지 객체를 로딩(지연 로딩)하는 구조임.


## 행위 패턴

### 책임 연쇄 패턴 (Chain of Responsibility)
- **개념(정의):** 요청을 처리할 수 있는 객체들을 체인(사슬) 형태로 묶고, 클라이언트의 요청이 들어오면 체인을 따라 차례대로 전달하며 자신이 처리할 수 있는 객체가 나타날 때까지 책임을 넘기는 패턴임.
- **목적과 필요성:** 요청을 보내는 객체와 이를 처리하는 객체 간의 결합도를 낮추고, 런타임에 요청을 처리할 객체의 순서를 동적으로 변경하거나 다중 필터링 처리를 유연하게 구현하기 위함임.
- **예시:** 웹 서비스의 로그인 요청 처리 시, 1단계: 인증 필터 -> 2단계: 로깅 필터 -> 3단계: 권한 필터 순으로 요청을 넘기며 검증하고, 중간에 통과하지 못하면 그대로 요청을 차단하는 자바 서블릿 필터(Filter) 구조임.

### 커맨드 패턴 (Command)
- **개념(정의):** 요청이나 명령 자체를 단순한 메서드 호출이 아닌, 하나의 독립된 객체로 캡슐화하여 표현하는 패턴임. 명령을 실행하는 인터페이스를 정의하고, 구체적인 명령 클래스들이 이를 구현함.
- **목적과 필요성:** 요청을 매개변수화하여 큐에 저장하거나 로그로 남길 수 있으며, 작업 취소(Undo) 및 재실행(Redo), 매크로 명령(여러 명령을 묶어 한 번에 실행) 기능을 시스템 구조 변경 없이 깔끔하게 구현하기 위함임.
- **예시:** 만능 리모컨의 버튼을 누를 때 램프 켜기 명령, TV 켜기 명령 등을 각각 `LightOnCommand`, `TvOnCommand` 객체로 캡슐화하여 리모컨 버튼 슬롯에 동적으로 갈아끼우고, 실행된 커맨드들을 스택에 쌓아 '실행 취소'를 구현하는 구조임.

### 인터프리터 패턴 (Interpreter)
- **개념(정의):** 특정 언어나 문법의 규칙을 정의하고, 이를 해석하는 구문 분석기(평가기)를 클래스 계층 구조로 확장성 있게 구성하는 패턴임. 문법의 각 규칙을 클래스로 정의하여 표현함.
- **목적과 필요성:** 간단한 미니 언어, 정규 표현식, 특정 포맷의 수식이나 쿼리를 자주 해석하고 평가해야 할 때, 문법 규칙을 쉽게 추가하고 변경할 수 있도록 구조화하기 위함임.
- **예시:** SQL 쿼리문 파서나 로마 숫자(예: IX = 9, VII = 7)를 아라비아 숫자로 변환하는 프로그램에서, 각 문법 규칙(1의 자리 규칙, 10의 자리 규칙)을 클래스로 정의해두고 문장을 순차적으로 해석하는 구문 분석 구조임.

### 이터레이터 패턴 (Iterator)
- **개념(정의):** 자료구조(배열, 리스트, 트리 등)의 내부 구현 방식과 상세 구조를 외부에 노출하지 않고, 그 안의 원소들을 순차적으로 접근(순회)할 수 있는 통일된 인터페이스를 제공하는 패턴임.
- **목적과 필요성:** 클라이언트가 컬렉션의 내부 저장 방식(연결 리스트인지 배열인지)에 상관없이 동일한 방식(`hasNext()`, `next()`)으로 안전하게 순회할 수 있도록 데이터 구조와 순회 로직을 격리하기 위함임.
- **예시:** 자바 컬렉션 프레임워크의 `ArrayList`와 `HashSet`이 내부 구조는 완전히 다르지만, 둘 다 `iterator()`를 호출하여 `while(iterator.hasNext())` 구조의 동일한 코드로 요소를 하나씩 꺼내 쓸 수 있게 만드는 방식임.

### 미디에이터 패턴 (Mediator)
- **개념(정의):** 여러 객체들 간의 복잡한 상호작용(M:N 관계)을 직접 소통하게 두지 않고, 하나의 중재자(Mediator) 객체에 집중시켜 모든 소통을 중재자가 통제하도록 캡슐화하는 패턴임.
- **목적과 필요성:** 객체들이 서로를 직접 참조하면 스파게티 코드가 되어 변경이 불가능해지므로, 중재자를 통해 느슨한 결합(1:1 관계들로 단순화)을 형성하여 각 객체의 재사용성을 높이기 위함임.
- **예시:** 공항의 수많은 비행기(객체)들이 충돌하지 않기 위해 서로 직접 무전하지 않고, 오직 중앙의 관제탑(중재자)과만 통신하며 이착륙 승인을 받아 안전하게 유기적으로 통제되는 구조임.

### 메멘토 패턴 (Memento)
- **개념(정의):** 객체의 내부 상태를 외부로 노출하여 캡슐화를 위배하지 않으면서도, 특정 시점의 객체 상태를 스냅샷 형태로 기록(저장)해두고 나중에 해당 상태로 안전하게 복구할 수 있도록 돕는 패턴임.
- **목적과 필요성:** 텍스트 에디터의 '실행 취소(Ctrl+Z)', 게임의 '체크포인트 저장' 기능처럼 상태를 이전으로 되돌려야 하는 기능을 객체의 프라이빗 속성을 훼손하지 않으면서 구현하기 위함임.
- **예시:** 바둑 게임 프로그램에서 사용자가 '무르기'를 요청했을 때, 바둑판의 현재 돌 위치 정보(내부 상태)를 `Memento` 객체에 저장해 과거 리스트에 넣어두었다가, 무르기 버튼 클릭 시 과거 메멘토를 복원하는 구조임.

### 옵저버 패턴 (Observer)
- **개념(정의):** 어떤 객체(Subject)의 상태가 변할 때, 그 객체에 의존하고 있는 다른 관찰자 객체(Observers)들에게 일제히 알림을 보내고 자동으로 상태를 업데이트하게 만드는 일대다(1:N) 의존성 패턴임.
- **목적과 필요성:** 이벤트 기반 시스템에서 상태 변화를 주기적으로 확인(Polling)하는 비효율을 없애고, 상태가 바뀐 시점에만 확실하게 통보받아 객체 간의 상태 동기화를 실시간으로 유지하기 위함임.
- **예시:** 유튜브 채널(Subject)을 구독한 수많은 유튜버 구독자(Observers)들이 있을 때, 채널에 새 동영상이 업로드되는 즉시 구독자들에게 자동으로 일제히 푸시 알림이 발송되는 구조임.

### 상태 패턴 (State)
- **개념(정의):** 객체의 내부 상태에 따라 행동(메서드의 동작)이 완전히 바뀌도록, 상태 자체를 하나의 클래스로 캡슐화하여 동적으로 행동을 교체하는 패턴임. 겉보기에는 객체의 클래스가 바뀌는 것처럼 보임.
- **목적과 필요성:** 자판기나 로그인 세션처럼 상태에 따라 수행할 행동이 복잡할 때, 메서드 내부에 무수히 나열되는 조건문(`if-else`, `switch`)을 제거하고 새로운 상태가 추가되어도 기존 상태 코드가 깨지지 않게 관리하기 위함임.
- **예시:** 뽑기 기계 클래스에서 동전이 없을 때 동전 반환 버튼을 누르면 '동전을 넣으세요'가 출력되고, 동전이 있을 때 누르면 '동전이 반환됩니다'가 출력되도록 `NoCoinState`와 `HasCoinState`라는 상태 클래스로 로직을 독립 분리한 구조임.

### 전략 패턴 (Strategy)
- **개념(정의):** 동일한 목적을 가진 알고리즘(전략) 패밀리를 각각 독립된 구체 클래스로 캡슐화하고, 실행(런타임) 중에 필요에 따라 컨텍스트에 주입하여 부품처럼 동적으로 교체·사용하는 패턴임.
- **목적과 필요성:** 메서드 내부에 분기문으로 알고리즘을 확장하는 구성을 탈피하여, 기존 클라이언트 코드를 수정하지 않고도 새로운 알고리즘(예: 배송비 계산 정책, 결제 수단 등)을 자유롭게 추가하고 갈아끼우기 위함임.
- **예시:** 쇼핑몰 결제 시스템에서 최종 결제 금액을 계산할 때, 컨텍스트 코드는 유지한 채 실행 중에 국외 배송 정책 전략(`InternationalShippingStrategy`) 혹은 국내 배송 정책 전략(`DomesticShippingStrategy`) 알고리즘 부품을 선택적으로 주입해 동작시키는 구조임.

### 템플릿 메서드 패턴 (Template Method)
- **개념(정의):** 부모 클래스에서 연산의 전체적인 알고리즘 뼈대(전체 실행 순서)를 메서드로 고정해 정의하고, 구체적으로 변하는 일부 단계의 구현은 서브클래스에서 오버라이딩하도록 미루는 패턴임. 상속을 기반으로 작동함.
- **목적과 필요성:** 전체적인 처리 흐름은 완벽히 동일하지만 세부 단계의 로직만 조금씩 다를 때, 전체 제어 흐름의 중복 코드를 부모 클래스에 모아서 일괄 관리하고 일부분만 안전하게 커스텀하기 위함임.
- **예시:** 음료 제조 클래스에서 1단계: 물 끓이기 -> 2단계: 우려내기 -> 3단계: 컵에 붓기라는 거대한 흐름은 상위 추상 클래스에 고정해 두고, 2단계의 '우려내기' 내용만 자식 클래스에서 각각 차 잎(Tea) 우려내기, 커피(Coffee) 추출하기로 오버라이딩하는 구조임.

### 비지터 패턴 (Visitor)
- **개념(정의):** 객체 구조(데이터 구조)와 그 구조 내의 원소들을 대상으로 수행할 실제 연산(비즈니스 로직)을 완전히 분리하는 패턴임. 데이터 구조 클래스 안을 바꾸지 않고, 방문자(Visitor) 객체가 돌아다니며 연산을 대신 수행함.
- **목적과 필요성:** 데이터 구조 자체는 거의 변하지 않지만 그 구조를 활용한 새로운 연산이나 통계 로직이 빈번하게 추가될 때, 데이터 원본 클래스의 구조를 전혀 더럽히지 않고 새로운 기능만 깔끔하게 추가하기 위함임.
- **예시:** 쇼핑 카트 안의 대량 상품들(데이터 구조)의 코드는 그대로 둔 채, 회원 등급별 할인율을 적용하는 방문자(`DiscountVisitor`)와 통계용 세금을 계산하는 방문자(`TaxVisitor`) 객체가 카트 내 상품들을 순회하며 각기 다른 연산을 수행하도록 만드는 구조임.



# 추상 팩토리 vs 팩토리 메서드

1. 팩토리 메서드 패턴 (Factory Method Pattern)

정의: 객체를 생성하는 인터페이스를 상위 클래스에 정의하되, 실제 어떤 구체 클래스의 인스턴스를 생성할지는 하위 서브클래스에서 결정하도록 미루는 패턴임.

핵심 메커니즘: 상속(Inheritance)을 기반으로 작동함. 부모 클래스에 추상 메서드로 팩토리 메서드를 열어두면, 자식 클래스가 이를 오버라이딩하여 원하는 구체 객체를 생성함.

초점: "하나의 제품(Product)"을 느슨한 결합으로 유연하게 생성하는 데 집중함.

2. 추상 팩토리 패턴 (Abstract Factory Pattern)

정의: 구체적인 제품 클래스를 직접 지정하지 않고, 연관되거나 의존적인 복수의 객체들의 제품군(Product Family) 전체를 한꺼번에 생성할 수 있는 상위 추상 인터페이스를 제공하는 패턴임.

핵심 메커니즘: 위임 및 합성(Composition)을 기반으로 작동함. 클라이언트 클래스가 추상 팩토리 인터페이스를 내부에 주입받아(Has-A), 그 팩토리 객체에 생성을 전적으로 위임함.

초점: 관련 있는 "여러 종류의 제품 세트(Product Family)"가 서로 짝이 맞도록 일관되게 생성하는 데 집중함.


팩토리 메서드: 가게 내부에서 "단 하나의 메서드가 구체적으로 어떤 제품 객체를 생성할지" 자식 클래스에 상속하는 하위 개념.

추상 팩토리 (↑): 여러 제품군을 아우르는 "거대한 제조 공장(가게 시스템) 자체를 통째로 위임·주입"하여 뒤섞임을 막는 상위 계층 개념.

# 전략 패턴 vs 상태 패턴 vs 방문자 패턴 
"인터페이스와 위임(Composition)을 활용해 if-else 분기문을 제거한다"는 강력한 구조적 공통점을 가지고 있어서 얼핏 보면 똑같아 보이지만, 누가 행동을 바꾸고(주체), 왜 바꾸는지(목적)가 완전히 다름.

**① 전략 패턴 (Strategy)**

"내가 원하는 부품을 직접 고르겠음"

동일한 목적을 가진 여러 알고리즘을 캡슐화하여, 필요에 따라 클라이언트가 직접 전략을 주입하는 패턴임.

비유 [쇼핑몰 결제]: 결제 창에서 사용자가 직접 [카카오페이], [네이버페이], [신용카드] 중 하나를 선택하면(클라이언트가 주입), 시스템은 그 전략에 맞춰 결제를 수행함.

특징: 컨텍스트(쇼핑몰)는 가만히 있고, 외부(사용자)가 전략 부품을 능동적으로 갈아끼움.

**② 상태 패턴 (State)**

"내 기분에 따라 내 행동이 자동으로 바뀜"

객체의 내부 상태가 변경될 때마다 객체의 행동이 자동으로 바뀌도록 상태 자체를 클래스화한 패턴임.

비유 [자판기]: 자판기에 돈이 없는 상태(NoCoinState)에서 반환 버튼을 누르면 "돈을 넣으세요"가 나오지만, 500원을 넣는 순간 자판기 스스로 상태를 HasCoinState로 바꾸고, 이제 반환 버튼을 누르면 "동전이 반환됩니다"로 행동이 바뀜.

특징: 외부 사용자는 버튼만 누를 뿐임. 객체 내부에서 전이(Transition)가 일어나 스스로 행동을 교체함.

**③ 방문자 패턴 (Visitor)** 

"손님이 집에 찾아와서 필요한 일을 해줌"

데이터 구조(클래스들)는 절대 변경하지 않으면서, 그 구조 안에 있는 원소들을 대상으로 수행할 새로운 연산(로직)만 외부 방문자 객체에 작성하는 패턴임.

비유 [장바구니 통계]: 마트 장바구니(데이터 구조) 안에 과일, 과자, 고기 클래스가 담겨있음. 상품 클래스 코드는 전혀 고치지 않고, 계산원(PriceVisitor)이 장바구니를 방문해 금액을 계산하고, 영양사(CalorieVisitor)가 방문해 총칼로리를 계산함.

특징: 데이터 구조는 가만히 있고, 외부의 연산자(Visitor)가 구조를 순회하며(더블 디스패치) 기능을 수행함.

"세 패턴 모두 인터페이스와 위임을 통해 내부 분기문을 제거한다는 구조적 공통점이 있다.
그러나 전략 패턴은 클라이언트가 필요에 따라 알고리즘 부품을 동적으로 갈아끼우는 것이 목적이고,
상태 패턴은 객체의 내부 상태 변화에 따라 스스로 행동을 전이시키는 것이 목적이며,
방문자 패턴은 데이터 구조를 변경하지 않고 새로운 연산 로직만 외부에 분리하여 추가하는 것이 목적인 점에서 결정적인 차이가 있다."
