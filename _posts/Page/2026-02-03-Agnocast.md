### 문제 정의

ComponentContainer 노드를 동일한 프로세스에 배치하면 zero copy 구현 완료 

하지만 fault isolation 문제 발생

왜냐하면 모든 기능을 하나의 노드에 모두 넣어버리면 하나의 기능에 문제가 생겼을 때, 다른기능도 같이 다운되어버릴 수 있음

그러나 또 기능을 분리해서 관리하자니 unsized type message 에 대한 제로카피 기법이 없음

기존에 이미 true zero copy communication 이 존재햇지만 **static size message**만 다룸 

**IceOryx단점**
- Memory Pool 방식을 사용하기에, 데이터 크기가 미리 정해져 있어야함 
- 가변데이터는 잘 처리하지 못함 (오히려 성능이 떨어지는 모습을 보임 )
- 유연했던 기존 코드를 하드코딩으로 배열 최대 크기 넣어서 수정해야함 
- => 데이터 크기 바뀔 때 마다 코드 수정 및 컴파일 필요 
- std::vector도 다 수정해야함 

**unsized message type에 대한 기존의 zero copy 기법**

**TZC**
- custom message type으로 구현해야함
- 배열 크기를 미리 할당해야함 

**LOT**
- requires conversin to custom message formats
- imposes library-specific coding constraint
- 개발 overhead 높임
- 코드 유연성 해침 
- std::vector::push_back 사용못함 (근데 ros개발자는 이 문법 즐겨 씀)

결국 위의 두 기술은 실무에 사용하기에 너무 난이도가 높고 복잡하여 사장되어버림 


이에 논문 저자는 ***Agnocast*** 라는 true zero-copy 프레임워크 제안함

- unsized message type도 지원
- ros2 표준(호환성) 해치지 않음
- 하드 코딩 없음 

사용법
- ROS2 pub/sub API 와 IPC 메세지를 위한 smart pointer type만 바꾸면 됨 

3가지 기술 제공
1. C++ 언어 환경과 IDL의 제약 규정 
2. LD_PRELOAD를 이용해서 heap 할당을 낚아채기 
3. Bridge를 통해 기존 노드와 Agnocast 노드 통신 지원

### 설계 요구사항 
1. [가장 중요] True zero-copy with **unsized message type**
- 프로세스간에 msg 통신 진행 시 serialization, deserialization을 포함해서 복사 비용이 발생하면 안됨 
- 모든 타입에 대한 zero copy 지원
2. do not need to rebuild tools, library
- zero copy를 사용하기 위해 기존 코드에 대한 과한 수정이 필요하게 되면 결국 기술이 사장됨
- adoption barrier를 피하자. 
3. apply only explicitly targeted paris of nodes
- zero copy가 필요한 부분에만 적용되도록 하자. 

그러면 저자가 말하는 unsized type message라는건 뭘까?

### [의문점] ROS2는 왜 직렬화를 해서 Data를 전달할까?

포인트 클라우드 msg로 예시를 들어보자. 

```cpp
struct PointCloud2{
    Header header;
    uint32_t header;
    uint32_t width;
    std::vector<uint8_t>data;
};
```

위와 같은 구조체가 있을 때, header, height, width는 stack/data영역에 저장되지만 **std::vector의 경우 가변길이**이므로 heap에 저장된다. 

구조체에는 vector 데이터가 아닌 heap에 저장된 vector의 포인터값만 들어있음 

그래서 구조체 크기만큼만 복사해서 보내면 이 데이터를 구독한 노드는 segmentaion Fault가 발생하게됨

### [의문점 2]왜 segFault가 발생하는가?

당연히 VM 주소가 다르니까. 

Process A와 Process B는 서로 독자적인 메모리 구조와 가상주소를 가지고 있음. 

그럼 Process A가 0x1000을 가리킬때 실제 메모리주소는 0x5000를 가리키고 Process B의 VM주소 0x1000은 0x6000을 가리킬 수도 있음

결국 같은 가상 주소값을 사용하는 것 처럼 보여도 실질적으로 매핑된 **물리 메모리 주소가 달라서** 이상한 값을 읽게 되는 것.

최종적으로 실질적인 데이터를 보내려면 실제 **heap data를 끄집어내고 Byte stream으로 Flattening**하는 작업, 즉 Serialization 과정이 필요하게 됨 


### 설계 제약사항 

inability **to specify custom memory allocators** for ROS2 message objects

zero copy를 구현하려면 msg가 무조건 바로 공유 메모리에 작성되도록 해야함.

iceOryx의 경우에는 static sized msg type을 사용하므로 필요한 메모리 미리 예측 가능 

반면, unsized msg의 경우, 런타임에 추가 메모리 할당이 발생하기도 함 

std::vector를 쓰면, 데이터를 넣다가(push_back) 자리가 부족해지면 실행 도중(Runtime) 갑자기 메모리를 새로 잡음 (Reallocation)

결국 기존 msg allocation 코드는 커스텀되어야함!!

하지만 ROS2의 msg allocation 코드는 이미 하드 코딩이 되어있음.

### 왜 allocation을 바꾸는 것이 실제로 적용되지 않는걸까?

내가 std::vector 코드에서 allocator 코드를 수정한다고 가정하자. 

이 수정한 코드는 vector가 일반 메모리가 아닌 공유 메모리에 할당되도록 했다고 한다. 

이렇게 std::vector 할당자를 수정해주어도 **rosidl** 시스템이 이를 사용하지 못하게 함
